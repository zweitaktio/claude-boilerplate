# i18n Rules

Rules for working with react-i18next. For full implementation details with React Router 7, see `vendor/react-router-7-i18n`.

## Rules

1. **All user-facing UI strings must use `t()`** — never hardcode display text

2. **Always provide English default** in every `t()` call:
   ```typescript
   t('nav.about', 'About')           // ✅ Good
   t('nav.about')                    // ❌ Bad — blank if key missing
   ```

3. **Never use dynamic keys** — `t()` must use static string literals:
   ```typescript
   t('status.active', 'Active')      // ✅ Static key
   t(`status.${value}`)              // ❌ Dynamic — parser can't extract
   t(statusKey)                      // ❌ Variable — parser can't extract
   ```

4. **Use lookup objects for data-to-translation mapping**:
   ```typescript
   // ✅ Good — static keys in lookup (inside component, after useTranslation)
   const STATUS_LABELS = {
     active: t('status.active', 'Active'),
     inactive: t('status.inactive', 'Inactive'),
   } as const
   const label = STATUS_LABELS[status]

   // ❌ Bad — dynamic key
   const label = t(`status.${status}`)
   ```

   **For keys defined outside components** (e.g., route configs), use a dummy extraction file:
   ```typescript
   // manual-translation-strings.ts — for extraction only, not runtime
   import { t } from '~/lib/t-dummy'  // No-op function
   t('route.products.name', 'Online Shop')
   t('route.about.name', 'About Us')
   ```
   The actual runtime call must still use `t()` with the static key.

5. **Run extraction after adding new keys**:
   ```bash
   yarn i18n:extract   # Extract keys from code
   yarn i18n:sync      # Regenerate EN from defaults (if configured)
   ```

6. **Never edit JSON translation files directly** — use extraction:
   ```typescript
   // ✅ Add t() call in code, then run yarn i18n:extract
   t('nav.newItem', 'New Item')

   // ❌ Don't manually add keys to JSON files
   ```

   **Exception:** If an existing English string changes, manually update ALL language files with translated versions of the new string.

7. **Use plurals correctly**:
   ```typescript
   // Translation file
   {
     "items_one": "{{count}} item",
     "items_other": "{{count}} items"
   }

   // Usage
   t('items', { count: 1 })   // → "1 item"
   t('items', { count: 5 })   // → "5 items"
   ```

8. **Avoid key name collisions** — parent and child keys must not share names:
   ```typescript
   // ❌ Bad — 'nav' used as both parent and key
   t('nav')        // Returns object, not string
   t('nav.about')

   // ✅ Good — unique parent name
   t('nav.title')
   t('nav.about')
   ```

9. **Use namespaces for feature separation**:
   ```typescript
   // Declare in route handle
   export const handle = { i18n: ['common', 'admin'] }

   // Use in component
   const { t } = useTranslation('admin')
   t('login.title', 'Admin Login')
   ```

10. **Use `<Trans>` for JSX inside translations** (links, formatting):
   ```typescript
   // ✅ Good — JSX embedded in translation
   <Trans i18nKey="terms.agreement">
     By signing up, you agree to our <Link to="/terms">Terms</Link>.
   </Trans>
   // Translation: "By signing up, you agree to our <0>Terms</0>."

   // ❌ Don't use Trans for plain text — use t() instead
   <Trans i18nKey="nav.about">About</Trans>  // Overkill
   ```

11. **Date formatting must use locale from i18n**:
    ```typescript
    import { format } from 'date-fns'
    import { de, enUS, fr } from 'date-fns/locale'

    const locales = { de, en: enUS, fr }

    // ✅ Always pass locale from i18n context
    format(date, 'dd.MM.yyyy', { locale: locales[currentLanguage] })

    // ❌ Never use native Date methods — no i18n support
    date.toLocaleDateString()
    new Intl.DateTimeFormat().format(date)

    // ❌ Never omit locale — formatting varies by language
    format(date, 'dd.MM.yyyy')
    ```
